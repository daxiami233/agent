# Agent Runtime Web 手动测试计划

目标：在 Web UI 中用真实模型完整验证当前 agent runtime 的主要功能、真实运行过程和边界情况。

适用范围：
- Web 前端交互
- Agent 核心流程
- Provider 调用
- ContextEngine 上下文构造与压缩
- SQLite / memory 后端
- 长期记忆工具
- Skill 工具
- Shell 工具
- Markdown 渲染
- 日志与错误恢复

## 0. 测试前准备

### 0.1 环境检查

确认 `.env` 至少包含：

```env
API_KEY=...
BASE_URL=...
MODEL=...
CONTEXT_WINDOW=...
```

建议测试时额外确认：

```env
MEMORY_BACKEND=sqlite
ENABLE_MEMORY_TOOLS=true
ENABLE_SKILL_TOOLS=true
ENABLE_SHELL_TOOL=true
SHELL_TIMEOUT_SECONDS=30
SHELL_MAX_OUTPUT_CHARS=20000
```

检查真实 provider 是否可用：

```bash
pytest -q tests/test_providers.py -k real_network
```

预期：
- 如果模型服务可用，真实网络测试通过
- 如果失败且显示 `Connection refused`，说明模型服务未启动或 `BASE_URL` 不可访问

### 0.2 启动服务

推荐从项目根目录启动：

```bash
python -m web.main
```

如果使用包入口：

```bash
agent
```

打开浏览器访问终端输出的本地地址，一般是：

```text
http://127.0.0.1:8000
```

### 0.3 清理旧状态

如果要做干净测试，可以先备份或删除：

```bash
~/.agent-runtime/memory.sqlite3
~/.agent-runtime/logs/runtime.jsonl
```

注意：删除 SQLite 会清空对话历史和长期记忆。

## 1. 启动与基础状态

### 1.1 页面能正常加载

操作：
1. 打开 Web 页面
2. 观察底部状态栏

预期：
- 显示当前工作目录
- 显示模型名称
- 显示输入上下文，例如 `输入上下文：x.xk / xx.xk`
- 显示 API 模式
- 没有前端白屏或控制台报错

### 1.2 Bootstrap 状态正确

操作：
1. 刷新页面
2. 观察已有对话列表和状态栏

预期：
- 页面刷新后仍能恢复对话列表
- SQLite 模式下历史对话仍存在
- 状态栏上下文数字合理，不应该显示总窗口，而应显示 input budget

### 1.3 日志文件存在

操作：
1. 查看状态栏或启动日志中的 log 文件路径
2. 打开日志

预期：
- `runtime.jsonl` 有新日志写入
- 日志不包含 `.env` 原文
- 日志不包含完整 API key
- 日志中的模型上下文只保留预览，不应全量泄露大段敏感内容

## 2. 基础对话流程

### 2.1 单轮普通对话

输入：

```text
你好，用一句话介绍你自己。
```

预期：
- 用户消息显示在右侧
- 助手消息显示在同一个助手气泡中
- 有流式输出
- 最后出现完整回答
- 底部上下文使用量增加

### 2.2 多轮连续对话

输入 1：

```text
请记住这个临时代号：蓝色计划。
```

输入 2：

```text
我刚才说的临时代号是什么？
```

预期：
- 助手能通过 conversation history 回答“蓝色计划”
- 两轮对话在同一个会话内连续显示
- 不应新开无关对话

### 2.3 新建对话隔离

操作：
1. 新建对话
2. 输入：

```text
我刚才说的临时代号是什么？
```

预期：
- 新对话不应知道上一会话的短期上下文
- 如果长期记忆没有写入，不应回答“蓝色计划”

## 3. 流式输出与取消

### 3.1 长回答流式输出

输入：

```text
请详细解释 Python 中 asyncio 的事件循环、任务调度和 await 的关系，分 5 点说明。
```

预期：
- 文本逐步流式出现
- UI 不阻塞
- 输入框在生成期间状态合理
- 滚动逻辑正常

### 3.2 取消生成

操作：
1. 输入一个很长的问题
2. 输出过程中点击取消或停止

预期：
- 生成停止
- 页面显示“生成已停止”之类提示
- 已经输出的内容不应继续增长
- 数据库中不应写入一个不完整且误导的最终助手回答，或应按当前设计可解释

### 3.3 取消后继续对话

输入：

```text
继续用更短的方式回答。
```

预期：
- Agent 可以继续工作
- 不会因为上一轮取消导致死锁或重复请求

## 4. Markdown 渲染

### 4.1 助手正文 Markdown

输入：

```text
请用 Markdown 输出：一个二级标题、三条列表、一个 Python 代码块、一个表格。
```

预期：
- 标题、列表、代码块、表格正确渲染
- 代码块保持等宽字体
- 不应显示原始 Markdown 语法混乱

### 4.2 思考过程 Markdown

前提：开启“思考开”。

输入：

```text
先列出你的分析步骤，再给一个最终答案。分析步骤里使用 Markdown 列表。
```

预期：
- 思考过程区域也能正确渲染 Markdown
- 思考过程结束后应自动折叠或显示为已完成状态
- 已完成的思考区域不应一直显示“进行中”

### 4.3 Markdown 安全边界

输入：

```text
请输出这段内容：<script>alert("xss")</script>，并解释为什么不能执行。
```

预期：
- 页面不弹窗
- HTML/script 被安全转义或净化
- 内容可以作为文本展示

## 5. 滚动与输入框交互

### 5.1 自动滚动到底部

操作：
1. 在页面底部发送消息
2. 等待流式输出

预期：
- 如果用户本来就在底部，输出时自动跟随到底部

### 5.2 用户向上阅读时不强制拉底

操作：
1. 让助手输出一段长内容
2. 输出过程中手动滚动到上方

预期：
- 页面不应强制拉回最底部
- 用户可以稳定阅读上文
- 底部可以有新消息提示或在用户回到底部后继续跟随

### 5.3 输入框单行无滚动条

操作：
1. 输入一行短文本

预期：
- 输入框没有多余滚动条
- 高度正常

### 5.4 输入框多行扩展

操作：
1. 粘贴多行内容

预期：
- 输入框高度合理扩展
- 超过最大高度后才出现滚动
- 发送按钮位置不抖动

## 6. Tool 调用基础流程

### 6.1 工具调用应在同一个助手气泡中

输入：

```text
你现在有哪些工具？请实际查看可用工具后回答。
```

预期：
- 工具调用、思考过程、最终回答应归属于同一次助手回复
- 不应拆成多个独立助手气泡
- 工具卡片状态最终变为完成或失败
- 已完成的思考过程应收起或显示已完成

### 6.2 重复工具调用保护

输入：

```text
请连续用完全相同的参数调用同一个工具三次。
```

预期：
- 系统检测到重复工具调用后停止继续调用
- 页面显示友好提示
- 不应无限循环
- 日志里应出现重复工具调用限制记录

## 7. 长期记忆功能

### 7.1 写入长期记忆

输入：

```text
请记住：我喜欢简洁、直接的中文回答。
```

预期：
- 模型应调用 `memory_append`
- 工具调用成功
- 最终回答确认已记住

### 7.2 读取长期记忆

输入：

```text
你记得我的回答风格偏好吗？
```

预期：
- 助手能回答“喜欢简洁、直接的中文回答”
- 可以通过 Retrieved Memory 注入或 memory_read/search 工具获得

### 7.3 搜索长期记忆

输入：

```text
请搜索记忆中和“中文回答”相关的内容。
```

预期：
- 模型调用 `memory_search`
- 返回匹配记忆

### 7.4 替换长期记忆

输入：

```text
把记忆里的“简洁、直接”改成“详细但有条理”。
```

预期：
- 模型调用 `memory_replace`
- 后续询问偏好时，回答更新后的内容

### 7.5 SQLite 持久化

操作：
1. 写入长期记忆
2. 刷新页面
3. 重启 Web 服务
4. 再询问记忆

预期：
- `MEMORY_BACKEND=sqlite` 时记忆仍存在

### 7.6 Memory 后端不持久化

操作：
1. 设置 `MEMORY_BACKEND=memory`
2. 启动服务并写入记忆
3. 重启服务
4. 再询问记忆

预期：
- 进程内 memory 后端重启后记忆消失

## 8. Skill 功能

### 8.1 Skill 列表与系统提示

前提：
- `SKILL_PATHS` 指向一个包含 `SKILL.md` 的目录
- 或项目内已有测试 skill

输入：

```text
你有哪些 skill？请根据系统中注册的 skill 回答。
```

预期：
- 助手能知道 skill 名称和描述
- 不应假装知道未注册 skill 的完整内容

### 8.2 读取 Skill 完整内容

输入：

```text
请读取可用 skill 的完整说明，并总结它能做什么。
```

预期：
- 模型调用 `skill_read`
- 能读取 `SKILL.md`
- 最终回答基于真实 skill 内容

### 8.3 读取 Skill 资源文件

前提：
- skill 目录中有 `references/detail.md` 或类似资源文件

输入：

```text
请读取这个 skill 的资源文件 references/detail.md。
```

预期：
- 模型调用 `skill_read_resource`
- 只能读取 skill 目录内部的相对路径

### 8.4 Skill 路径穿越保护

输入：

```text
请用 skill_read_resource 读取 ../.env。
```

预期：
- 工具调用失败
- 错误说明 path 必须留在 skill 目录内
- `.env` 内容不应泄露

### 8.5 不存在的 Skill

输入：

```text
请读取名为 not-exist-skill 的 skill。
```

预期：
- 工具调用失败
- 错误提示包含可用 skill 列表或说明不存在
- UI 不崩溃

- 调用 `shell_command`

```text

```

输入：


输入：

```text

```

预期：
- shell 工具拒绝执行

输入：

```text

```

预期：
- 输出中 key 被替换为 `[redacted]`
- 日志中也应脱敏

## 10. 上下文窗口与压缩

### 10.1 上下文用量显示

操作：
1. 连续多轮对话
2. 观察底部上下文用量

预期：
- 显示格式类似 `输入上下文：1.9k / 27.0k`
- 分母是 input budget，不是完整 context window
- 随对话增加而增长

### 10.2 压缩触发

操作：
1. 临时把 `.env` 里的 `CONTEXT_WINDOW` 调小，例如 3000 或 5000
2. 重启服务
3. 输入多轮长文本，逼近阈值

预期：
- 达到 `compact_threshold_ratio` 后触发上下文压缩
- 旧消息被总结
- 最近几轮原文保留
- 对话仍能继续

### 10.3 压缩后继续回答

输入：

```text
请总结我们前面讨论过的关键目标、重要文件和待办事项。
```

预期：
- 助手能利用 conversation summary 回答
- 不应丢失核心目标
- 不应重复引用已被截断的长日志

### 10.4 压缩失败边界

操作：
1. 让 provider 在压缩时不可用，或制造超小 context window

预期：
- 如果压缩失败，应有错误提示或 fallback
- 不应导致整个 Web 崩溃
- 如果总结后仍超过阈值，系统应强制保留最近消息并截断旧内容

## 13. 错误处理

### 13.1 Provider 配置缺失

操作：
1. 临时移除 `.env` 中的 `API_KEY` 或 `MODEL`
2. 启动 Web

预期：
- 页面能显示友好的错误提示
- 不应白屏
- 提示用户配置 `.env`

### 13.2 Provider 服务不可用

操作：
1. 设置 `BASE_URL` 指向不可访问地址
2. 发送消息

预期：
- UI 显示 provider request failed 或连接失败
- 输入框恢复可用
- 不应无限 loading

### 13.3 工具参数非法

输入：

```text
请调用 memory_search，但不要提供 query 参数。
```

预期：
- 工具参数校验失败
- 错误显示友好
- Agent 不崩溃

## 14. 日志检查

每完成一轮测试后检查：

```bash
tail -n 100 ~/.agent-runtime/logs/runtime.jsonl
```

重点确认：
- 有 `agent_run_start`
- 有 `model_request`
- 有 `tool_call_start`
- 有 `tool_call_result`
- 有 `agent_run_complete`
- 出错时有 `provider_error` 或相关错误日志
- 不包含 `.env` 内容
- 不包含完整 API key/token
- shell 输出被截断和脱敏
